ПОЛНОЕ ТЕХНИЧЕСКОЕ ЗАДАНИЕ
Подтверждение компаний исполнителей, групповые функции и чаты
1. Общая модель платформы

На платформе существуют две стороны:

Заказчик

Исполнитель

И заказчик, и исполнитель могут иметь статус:

физическое лицо

самозанятый

индивидуальный предприниматель (ИП)

юридическое лицо (ООО)

Групповые функции реализуются ТОЛЬКО для исполнителей.

Заказчики:

не имеют команд;

не управляют составом исполнителей;

взаимодействуют с исполнителями как с единым субъектом.

2. Принцип доверия и прав доступа

Выбор статуса (ООО / ИП / самозанятый / физлицо) является декларативным.

Статус сам по себе не даёт дополнительных прав.

Доступ к расширенному функционалу предоставляется только после подтверждения.

Подтверждение осуществляется автоматически, без ручной модерации и документов.

3. Подтверждение компании исполнителя
3.1 Этап 1 — Подтверждение существования компании

Исполнитель выбирает статус ИП или ООО.

Исполнитель вводит ИНН.

Платформа выполняет автоматическую проверку ИНН через ФНС.

Подтверждение считается успешным, если:

компания найдена;

тип компании соответствует выбранному статусу;

компания имеет действующий статус.

После успешной проверки фиксируется факт существования компании.

Подтверждение существования компании:

не подтверждает полномочия пользователя;

не даёт доступа к групповым функциям.

3.2 Этап 2 — Подтверждение связи пользователя с компанией

Подтверждение связи требуется только для доступа к групповым функциям.

При попытке использовать групповые функции платформа предлагает подтвердить компанию.

Исполнитель вводит корпоративную почту, связанную с доменом компании.

Платформа отправляет письмо с подтверждением на указанную почту.

Пользователь подтверждает почту по ссылке из письма.

После подтверждения фиксируется связь пользователя с компанией.

Корпоративная почта:

не используется для входа в аккаунт;

используется только для подтверждения полномочий.

Если у компании отсутствует корпоративная почта, групповые функции недоступны.

4. Кто может использовать групповые функции

Групповые функции доступны ТОЛЬКО при одновременном выполнении условий:

Пользователь находится в роли исполнителя.

Статус исполнителя — ИП или ООО.

Подтверждено существование компании (ИНН).

Подтверждена связь пользователя с компанией (корпоративная почта).

Физические лица и самозанятые не имеют доступа к групповым функциям.

5. Команды исполнителей
5.1 Понятие команды

Команда — это:

группа исполнителей;

принадлежащая одной подтверждённой компании-исполнителю;

действующая как единый исполнитель для заказчика.

5.2 Создание команды

Команду может создать только подтверждённый исполнитель.

При создании команды:

указывается название команды;

создаётся внутренний командный чат;

создатель становится администратором команды.

5.3 Роли в команде

Минимальный набор ролей:

Администратор команды:

добавляет и удаляет исполнителей;

управляет командным чатом;

назначает команду на задачи.

Исполнитель команды:

участвует во внутреннем чате;

участвует в задачах команды.

6. Добавление исполнителей в команду

Добавлять исполнителей может только администратор команды.

Добавление осуществляется по приглашению.

Приглашённый пользователь должен принять приглашение.

Без подтверждения приглашения доступ не предоставляется.

6.1 Ограничения

Обязательные лимиты:

ограничение на количество исполнителей в команде;

ограничение на количество приглашений за период;

ограничение на количество команд.

Лимиты управляются централизованно.

7. Задачи и назначение исполнителей
7.1 Назначение задачи

Задача может быть назначена:

одному исполнителю;

команде исполнителей.

Для заказчика команда отображается как один исполнитель.

7.2 Поведение команды в задаче

Если задача назначена команде:

все участники команды получают доступ к чату задачи;

заказчик общается с командой через чат задачи;

внутренние обсуждения ведутся в отдельном командном чате.

8. Чаты и единый интерфейс общения
8.1 Типы чатов

Внутренний чат команды

доступен только исполнителям команды;

заказчик не имеет доступа;

используется для внутренней координации.

Чат задачи

доступен заказчику и исполнителям;

используется для общения по задаче;

является официальным каналом коммуникации.

8.2 Единая страница чатов

У исполнителя существует одна страница чатов.

На странице реализовано явное переключение между:

внутренним чатом команды;

чатом конкретной задачи.

Переключение происходит без перезагрузки страницы.

8.3 Обозначение контекста (обязательно)

Пользователь всегда должен понимать, где он пишет:

Внутренний чат команды:

пометка «Внутренний чат команды»

указание, что сообщения видны только команде

Чат задачи:

указание номера / названия задачи

указание, что сообщения видны заказчику

8.4 Поведение сообщений

Поле ввода единое, но привязано к текущему контексту.

Невозможно отправить сообщение «не в тот чат».

При смене контекста история сообщений меняется.

9. Уведомления

Система уведомляет:

о приглашении в команду;

о назначении на задачу;

о новых сообщениях;

при этом различает:

внутренние чаты;

чаты задач.

10. Защита от абуза

Обязательные меры:

Групповые функции недоступны без подтверждения компании.

Нельзя добавлять пользователей без их согласия.

Ограничены массовые приглашения.

Логируются действия администраторов команд.

Администрация вправе:

отключать команды;

аннулировать чаты;

ограничивать доступ при нарушениях.

11. Политика и правила платформы

В правила платформы необходимо включить положения:

Пользователь обязуется указывать достоверные сведения о своём статусе
и не выдавать себя за представителя компании без законных оснований.

Групповые и командные функции предназначены исключительно
для совместной работы исполнителей при выполнении задач.

В случае выявления недостоверных сведений или злоупотреблений
администрация платформы вправе ограничить доступ к функциям,
приостановить или заблокировать аккаунт,
а также аннулировать созданные команды и чаты
без предварительного уведомления.

12. Критерии готовности

Функционал считается реализованным, если:

подтверждение компании осуществляется автоматически;

групповые функции доступны только исполнителям ИП / ООО;

заказчики не имеют командных функций;

команды имеют внутренние чаты;

задачи имеют отдельные чаты;

существует единый интерфейс переключения чатов;

архитектура исключает массовый абуз.
